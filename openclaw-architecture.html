01 / Background

背景与问题

当 AI 助手从单一对话工具演进为多实例协作体系，传统的"一问一答"模式暴露出哪些根本性问题？

!

上下文断裂

不同 AI 工具之间无法共享记忆。Claude Code 不知道 OpenClaw 讨论过什么，Telegram 指令的上下文在会话结束后丢失，每次都需要从头说明背景。

?

职责边界模糊

多实例并存时，缺乏明确的路由表和分工矩阵。「这个任务该交给谁？」成为反复出现的决策负担，导致任务冲突或遗漏。

</>

代码能力与日常能力割裂

OpenClaw 擅长日常对话和调度编排，但重度代码开发力不从心；Claude Code 代码能力强，但缺乏业务上下文和调度能力。两者能力互补却难以协作。

#

跨实例状态不可见

四个实例分布在三台机器上，Will 无法一眼看到「现在每个实例在做什么」。任务超时、委派失败、重复劳动频发。

核心洞察：我们需要的不是"更强的单一 AI"，而是一个 共享记忆 + 明确分工 + 自动路由 的多实例协作体系。 不是工程流水线，而是人格化搭档团队。

03 / Collaboration Layers

三层协作机制

Telegram 负责人机交互，OpenClaw 负责智能编排，Claude Code 负责深度执行 —— 三层各司其职，通过 clawmem 共享上下文。

T

Telegram 交互层

人机界面

Will 的主要入口。通过 Telegram Bot 发送自然语言指令，接收任务结果和状态通知。

  • 自然语言指令 → OpenClaw 解析执行
  • 图片/文件上传 → 自动识别处理
  • 实时状态推送 → 任务进度通知
  • 多 Chat 分流 → 不同实例不同频道
  • 语音消息 → 转文字后理解执行
O

OpenClaw 编排层

智能调度

四实例组成的 AI 搭档团队。根据路由表自动分配任务，管理跨实例协作和状态同步。

  • 路由表驱动 → 按业务+能力自动分配
  • 蜂群编排 → 子 Agent 并行执行
  • 轻量代码修改 → Edit 工具直接改
  • 跨实例委派 → 任务看板状态追踪
  • Cron 定时任务 → SNS/监控自动化
  • Skills 体系 → 可复用能力封装
C

Claude Code 执行层

深度编码

Opus 4.6 驱动的重度代码执行引擎。处理新功能开发、大规模重构、CI/CD 等需要全项目理解的任务。

  • 新功能开发 → Opus 代码能力强
  • 大规模重构 → 全项目理解 + 原子提交
  • OpenClaw 配置管理 → JSON 编辑 + 热重载
  • OpenClaw 升级 → npm install + SIGUSR1
  • PR 审查 → 深度代码理解
  • GSD 工作流 → 计划/执行/验证闭环
协作核心原则：不越权 · 不抢功 · 不甩锅 · 要闭环。 每个实例都是人格化搭档，不是工程流水线上的工位。按路由表行事，委派必须给足上下文，接收必须回报结果。
04 / Workflow Scenarios

四个典型工作流场景

从需求到交付，不同场景下三方如何无缝协作——覆盖功能开发、系统升级、SNS运营、直接编码四种核心场景。

01
新功能开发
Telegram → OpenClaw → Claude Code → 交付
  1. Will 通过 Telegram 告诉 ユキ："猫舍网站加一个预约表单"
  2. ユキ 分析需求，拆解为前端+后端+数据库三个子任务
  3. ユキ 确认方案后，通过 coding-agent Skill 调度 Claude Code
  4. Claude Code 从 clawmem 读取项目上下文，了解现有架构
  5. Claude Code 实现功能，原子提交，运行测试
  6. ユキ 收到完成通知，验证结果后通过 Telegram 告知 Will
02
OpenClaw 系统升级
版本检查 → 四实例滚动升级 → 验证
  1. Will 说："升级一下 OpenClaw"
  2. ユキ 检查 npm registry 版本差异，评估升级风险
  3. ユキ 确认后调度 Claude Code 执行升级脚本
  4. Claude Code 在本机执行 npm install -g openclaw@latest
  5. Claude Code 通过 SSH 升级远程实例（ハル@M2, アキ@MBP）
  6. Claude Code 发送 SIGUSR1 热重载信号，逐一验证四实例状态
03
SNS 自动运营
定时触发 → 内容生成 → 自动发布
  1. Cron 定时器 在预设时间触发 ナツ 的 SNS 任务
  2. ナツ 从 shared-knowledge 读取 SNS 策略和品牌指南
  3. ナツ 编排蜂群子 Agent 并行生成多平台内容
  4. 蜂群 Agent 生成文案 → ナツ 质量校验（字数/敬语/品牌名）
  5. ナツ 通过 API 自动发布到各社交平台
  6. ナツ 将发布结果汇报到 Telegram，Will 异步查看
04
直接终端编码
CLI → clawmem 自动注入 → 开发 → 记忆沉淀
  1. Will 在终端直接输入 cd ~/projects/xxx && claude
  2. clawmem SessionStart Hook 自动注入相关项目记忆
  3. Claude Code 带着完整上下文开始开发工作
  4. Will 与 Claude Code 交互式完成复杂编码任务
  5. clawmem Stop Hook 自动提取决策、生成 handoff 文档
  6. 下次 OpenClaw 或 Claude Code 会话都能召回本次决策
05 / Memory Bridge

clawmem 记忆桥梁

clawmem 是整个协作体系的"大脑"——它让 OpenClaw 的业务知识和 Claude Code 的开发上下文实现双向流通，消除上下文断裂。

四种记忆类型

类型用途触发时机
user用户偏好、习惯偏好信号确认后
feedback纠错、反馈、已验证方法用户纠正或确认时
project项目状态、决策、死线架构决策 / 里程碑变更
reference工具文档、API手册首次接触新工具/SDK

混合检索管线

Query 输入
→
自动路由
→
BM25 + 向量
→
Top-K 返回
  • "为什么我们选了 X" → 因果图遍历
  • "上次会话做了什么" → 会话历史
  • "和 X 相关的还有什么" → 向量近邻
  • 复杂多主题查询 → 并行分解

六大生命周期 Hooks —— 全自动记忆读写

  • SessionStart
    启动时注入上下文 — postcompact-inject + curator-nudge，确保新会话即刻获得相关记忆
  • UserPromptSubmit
    用户输入时 — context-surfacing 自动召回相关记忆，为每次回答注入背景知识
  • PreCompact
    上下文压缩前 — precompact-extract 提取关键信息，防止长对话中的重要决策丢失
  • Stop (Decision)
    会话结束 — decision-extractor 自动保存本次会话中的架构决策和技术选型
  • Stop (Handoff)
    会话结束 — handoff-generator 生成交接文档，下次会话可完美续接未完成的工作
  • Stop (Feedback)
    会话结束 — feedback-loop 收集纠错信号，持续改进记忆质量和推荐准确度

数据流转全景

07 / Information Sync

信息同步机制

四实例分布在三台机器上，信息同步是协作的命脉。三种同步机制覆盖不同时效需求。

R 实时共享

  • task-board.md 任务看板 — 三实例直接读写同一文件（本地文件系统）
  • routing-table.md 路由表 — 职责边界的 Single Source of Truth
  • clawmem MCP Server — Claude Code 每次输入实时检索
  • Telegram 消息 — Will 的指令即时到达目标实例

S 定期同步

  • cross-instance-sync.py — 每日 04:15 自动同步共享知识库
  • Skills 同步 — rsync 从 ユキ → ナツ/ハル/アキ
  • clawmem collection update — 定期重建向量索引
  • Heartbeat 检查 — 各实例周期检查委派任务

E 事件驱动

  • SIGUSR1 热重载 — 配置变更后立即生效，无需重启
  • Stop Hook → handoff — 会话结束时自动生成交接文档
  • 委派协议 — 跨实例任务在看板中状态流转
  • 错误熔断 — 3次失败自动标记 BLOCKED

Skills 跨实例同步流程

08 / Future Roadmap

未来扩展

从当前的四实例协作到更智能的自主协作体系——每个阶段都在解决一个具体的协作痛点。

Phase 1 — 可视化

监控 Dashboard

在 OpenClaw Web 界面展示四实例实时状态、任务看板全景、资源使用率和错误日志。 让 Will 一个页面看到所有实例在做什么，不再需要逐个检查。 包含 Heartbeat 心跳监控和任务超时自动告警。

Web UI实时监控告警推送状态聚合
Phase 2 — 智能路由

委派智能路由 + 自动熔断

基于路由表的静态分配升级为动态智能路由。根据实例当前负载、能力匹配度、历史成功率 自动推荐最佳委派目标。集成 3 次错误熔断机制，防止任务无限重试浪费资源。 任务看板与熔断联动，BLOCKED 状态自动升级通知。

动态路由负载均衡熔断机制自愈能力
Phase 3 — 记忆增强

跨实例语义记忆搜索

将 clawmem 的能力扩展到所有实例。任何实例都能语义检索全局知识图谱， 不仅包括 shared-knowledge 的文件，还包括每个实例的对话历史、决策记录和反馈信号。 支持因果链追溯："这个设计决策是什么时候、谁、基于什么原因做的？"

语义检索知识图谱因果追溯全局记忆
Phase 4 — 自主协作

产出质量校验 + 自主工作流

引入产出质量校验管线：SNS 文案自动检查字数、品牌名、禁用词；客服回复验证敬语体； 技术方案强制包含前提/步骤/风险三章节。校验通过后自动发布，不通过则自动修正重试。 最终实现从需求输入到交付的全自主工作流，Will 只需审批关键节点。

质量校验自动修正自主执行审批网关
