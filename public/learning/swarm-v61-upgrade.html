核心挑战与解决方案

蜂群引擎 v6.0 的 SKILL.md 已经 420 行，orchestrator-prompt-template 1800 行，references 引用 26 个文件。它是一个单体巨兽——调研、开发、内容、审查所有场景的逻辑混在一起。每次改动都像给巨人缝补丁。

而且 OpenClaw 的 skill 触发靠 description 关键词匹配，模型弱的时候（如 Kimi K2.5 做日常对话）扫 25+ 个 skill 的 description 就已经很吃力，再增加 skill 数量只会让路由更不准。

v6.1 核心理念：一个入口 + 内部分流

不增加 skill 数量。对外只有一个触发入口（swarm-dev-engine），Commander 内部根据任务类型自动加载不同的 profile 配置。调研、开发、内容三种场景各自有独立的配置，但对外是统一的。

Gate 类型

  • auto — 上一阶段完成且无 BLOCKER 自动继续
  • manual — 发 Telegram 等 Will 确认
  • loop_check — 检查条件决定是否回退
  • quality_gate — 检查产出质量指标

发现的问题

  • 2个 TODO 残留 — no-placeholder 铁律未生效
  • Discovery 机制 — 完全没用起来

根本原因：Orchestrator 没正确注入 v6.0 新铁律给 Worker。产出量和结构上去了，但新能力的注入管道需要加强。

5 条关键学习

1
不要把一个 skill 做得无限大 — 分场景 profile 比分 skill 更轻量，维护成本更低。
2
路由问题靠 description 改进 — 不需要新的路由系统，优化现有 description 的清晰度即可。
3
弱模型也要能用 — 一个入口自动分流比让模型选 skill 更可靠，降低对模型能力的依赖。
4
主动告知比等用户问更有价值 — Claude Code 的 Plan Mode 思路，AI 应该主动说明自己能做什么。
5
验证要看到新能力是否真正生效 — ナツ测试暴露了注入管道的问题，测试不能只看产出量。

主动告知铁律

写入 AGENTS.md 的新规则：所有模型（包括 Kimi 默认模型）必须主动告诉用户有什么 skill 可用、建议怎么做。

💡 我有 [skill名]，可以 [做什么]。要用吗？

不主动告知 = 浪费用户时间去猜 AI 有什么能力。

技术细节

Profile 配置文件结构

每个 profile 是一个独立的 markdown 文件，位于 references/ 目录：

  • profile-research.md (1.2KB) — 调研场景配置
  • profile-dev.md (1.6KB) — 开发场景配置
  • profile-content.md (1.5KB) — 内容场景配置

Commander 根据任务类型动态加载对应 profile，注入到 Worker 的 system prompt 中。

Cycle 引擎实现

Cycle 引擎通过 cycle-plan.json 定义任务流程：

{
  "mode": "serial" | "parallel" | "loop",
  "phases": [...],
  "gates": [...],
  "max_iterations": 3
}

Gate 支持四种类型：auto、manual、loop_check、quality_gate。阶段间产出通过上下文传递。

v6.0 vs v6.1 架构对比

v6.0（单体架构）：

  • 一个巨大的 orchestrator prompt（1800行）
  • 所有场景逻辑混在一起
  • 修改一处可能影响其他场景

v6.1（场景化架构）：

  • Commander 轻量核心 + 可插拔 Profile
  • 调研/开发/内容各自独立配置
  • 修改一个 profile 不影响其他场景
