1Harness Engineering 六层架构全景对比

Harness Engineering 将 Agent 系统抽象为六个层级（L1-L6），从上层的上下文信息边界到底层的约束校验与失败恢复。以下对比展示了 OpenClaw 作为完整 Harness 与 Hermes/Honcho 作为记忆子系统的结构性差异。

L6
OpenClaw

PUA-Lite 三条红线 + 五步方法论 + 失败升级机制。含 circuit breaker 熔断与闭环验证。约束层是生产级 Harness 的底线。

Hermes / Honcho

未见明确设计。Honcho 本身不处理行为约束或失败恢复逻辑，这些属于上层 Harness 的职责。

L5
OpenClaw

heartbeat + watchdog-pair 双实例互监、PUA-Lite 闭环验证、蜂群独立 Review Agent。观测与评估体系已验证。

Hermes / Honcho

未见明确设计。作为记忆层，不原生提供系统级观测能力。

L4
OpenClaw

memory-lancedb v2.0、daily notes、procedures、shared-knowledge + 四实例同步。但实现偏文件化，检索架构有优化空间。

Hermes / Honcho

强项。Honcho 核心即结构化记忆，原生支持 session-level + user-level state 管理，1024维向量 + BM25+rerank 混合检索。

L3
OpenClaw

强项。蜂群引擎（Swarm Engine）、subagent 编排、cron 调度、ClawFlow、auto-pipeline / auto-dev-loop 均已生产验证。

Hermes / Honcho

未明确。relay 层目前看起来主要是请求转发，不具备复杂的执行编排能力。

L2
OpenClaw

非常丰富。MCP、browser automation（playwright-cli / CDP）、Peekaboo、多 channel 插件（Telegram / Discord / Lark）。

Hermes / Honcho

几乎未见。Hermes 的侧重点是 memory relay，没有 expose 出丰富的工具调用层。

L1
OpenClaw

成熟。Skills 渐进式披露、System Prompt 分层、LCM 上下文压缩。但在检索精度上仍有提升空间。

Hermes / Honcho

强项。Honcho 专为上下文和记忆管理设计，BM25 rerank、1024维向量，检索召回质量高。

各层详细分析

L1 Context 信息边界 — 谁的上下文管理更强？

OpenClaw 在 L1 层面已经相当成熟。通过 Skills 渐进式披露机制，系统可以按需加载能力模块；System Prompt 分层设计保证了不同场景下的提示词隔离；LCM（Lossless Context Management）上下文压缩技术则在长对话场景下保持了内存可控。

然而，Hermes/Honcho 的优势在于其专门为上下文管理设计了一套检索增强体系：1024维向量 + BM25+rerank 混合检索。这意味着在「从历史记忆中召回相关信息」这一具体任务上，Hermes 的召回质量理论上更高。OpenClaw 当前的记忆检索主要依赖 LanceDB 的向量搜索，尚未引入 BM25 和 rerank。

关键洞察：OpenClaw 的 L1 是「系统的、工程化的」，Hermes 的 L1 是「专业的、聚焦的」。前者覆盖全面，后者在单点上有技术优势。

L2 工具系统 — Hermes 的明显短板

这是 OpenClaw 与 Hermes 差距最大的一层。OpenClaw 拥有丰富的工具生态：

  • MCP：Model Context Protocol，统一工具调用接口
  • Browser Automation：playwright-cli、CDP、Lightpanda 三轨道方案
  • Peekaboo：macOS UI 自动化
  • Channel 插件：Telegram、Discord、Lark（飞书）等多平台集成

Hermes 目前构建在 localhost:8000 的 Honcho 记忆架构 + localhost:4100 的 provider relay 之上，其核心定位是记忆层，没有 expose 出对等的工具系统。如果 Hermes 要独立发展为完整 Harness，L2 需要从 0 开始建设。

L3 执行编排 — 蜂群引擎的护城河

OpenClaw 的执行编排能力是其核心竞争力之一：

  • Swarm Engine 蜂群引擎：多 agent 并行执行 + 交叉验证 + 独立审查
  • Subagent 机制：任务可以递归分解到子 agent
  • Cron 调度：定时任务自动化
  • ClawFlow：工作流编排引擎
  • Auto-pipeline / Auto-dev-loop：非代码批量任务和代码开发的自动化闭环

Hermes 目前没有看到对等的执行编排设计。其 relay 层（localhost:4100）更像是模型请求的转发适配器，而非任务编排中枢。

L4 记忆与状态 — Hermes 的主场

这是 Hermes/Honcho 的设计原点，也是其最强的一层：

  • 1024维向量：更高的语义表达能力
  • MiniMax / Kimi embeddings 双兜底：模型可用性保障
  • BM25 + rerank 混合检索：关键词召回 + 语义重排序，稳定性优于纯向量检索
  • Session-level + User-level state：结构化记忆管理

OpenClaw 的记忆体系包括 memory-lancedb v2.0、daily notes、procedures、shared-knowledge 和四实例同步，但实现方式偏文件化，检索架构相对简单。这也是 DeepSeek 指出「精致 L4 是技术幻觉」的语境——记忆层做得再好，也不能掩盖其他层级的不足。

L5 评估与观测 — OpenClaw 的成熟体系

OpenClaw 在生产环境中运行了一套完整的观测体系：

  • Heartbeat：各实例定时健康检查
  • Watchdog-pair：ユキ/ナツ/ハル/アキ 四实例互相监控
  • PUA-Lite 闭环验证：行为结果必须通过验证命令确认
  • 蜂群 Review Agent：多模型交叉验证的独立审查机制

Hermes 作为新兴项目，尚未建立起同等级别的观测能力。

L6 约束/校验/失败恢复 — Harness 的底线

OpenClaw 的 PUA-Lite 框架提供了系统级的约束机制：

  • 三条红线：闭环验证、事实驱动、禁止原地打转
  • 五步方法论：主动执行、卡住升级、子 agent 验证等
  • Circuit breaker：连续失败后的自动熔断保护

Hermes/Honcho 的定位是基础设施层，安全约束需要上层 Harness 来保障。

📌 基础结论

OpenClaw 是覆盖 L1-L6 的完整 Harness；Hermes 更像是优秀的 Memory/Context 子系统。两者在架构定位上有本质区别。

工程落地 / 成本 / 风险视角
OpenClaw 具有压倒性优势（25+ Skills、四实例、watchdog 均已生产验证）；Hermes 若替代 OpenClaw，业务停摆风险极高。

  • 替代风险：🔴 极高风险 Hermes 替代 OpenClaw 将导致 25+ Skills 归零、蜂群引擎报废、四实例架构断裂、福楽キャッテリー等品牌业务至少停摆 2-4 周。

GLM-5 结论：OpenClaw 为主，Hermes 仅做边缘轻量试点。不要在核心业务上赌未经验证的组件。

Qwen3.5

技术架构 / 演进趋势视角
Hermes 是 L1 Context 层的专用记忆系统，OpenClaw 是覆盖 L1~L6 的完整 Harness。两者不是同一维度的竞争关系。

Qwen3.5 结论：OpenClaw 是躯体，Hermes 是记忆神经。两者整合，而不是二选一。Hermes 最可能的路径是成为 OpenClaw 的「记忆后端」。

DS

DeepSeek Thinking

Harness 工程深层思辨视角
"精致 L4" 是一种技术幻觉。记忆层做得精致最危险之处在于，它会让人产生"系统很成熟"的错觉。

DeepSeek 提出了 Harness Engineering 领域一个常被忽视的洞察：Agent 的失败模式几乎从不来自模型不够聪明，而总是来自 Harness 的某个环节断裂。 L4 过于精致可能会掩盖 L2/L3/L5/L6 的实际粗糙。

同时，DeepSeek 观察到 OpenAI 和 Anthropic 在记忆上的权重都很克制。以 Claude Code 为例，它采用「极其克制的记忆策略」——仅在 .claude 项目文件中写入结构化记忆，而不构建通用会话间记忆系统。这反映了一种工程哲学：

记忆应该是确定性的、文件化的、人类可审计的。

OpenAI 和 Anthropic 将绝大部分工程资源花在工具调用、错误恢复、安全沙箱上，而非记忆层。这与当前市场过度追捧「长记忆」的风气形成鲜明对比。

记忆是 Agent 的面子，工具和错误处理才是里子。先把里子做好，再考虑面子升级。

— DeepSeek Thinking

3核心争议与思辨

子议题一：L4 精致化是否是陷阱？

DeepSeek 的核心论点可以归纳为：在一个不完整的 Harness 中，过度投资于 L4 记忆层会产生技术幻觉。

具体来说：

  • 「这个 Agent 记得三个月前的事」在演示中极具说服力
  • 观众会因此推断「这个系统很成熟」
  • 但实际上，系统的失败可能发生在工具调用（L2）、执行编排（L3）、观测评估（L5）或安全约束（L6）中的任何一环

这并不意味着记忆层不重要，而是说 investing 的优先级应该遵循 Harness 的依赖关系：底层（L6 → L5 → L3 → L2）比上层（L1 → L4）更关键。没有稳固的工具、执行、观测和约束系统，再精致的记忆也不过是空中楼阁。

子议题二：记忆在 OpenAI/Anthropic 的 Harness 里占多大权重？

DeepSeek 对 Claude Code 的观察提供了一个重要的行业参照。Claude Code 的记忆策略极其克制：

  • 没有通用会话间记忆系统
  • 仅在项目文件中写入结构化记忆（.claude 目录）
  • 记忆是确定性的、文件化的、人类可审计的

这反映了一种工程优先级：OpenAI 和 Anthropic 将绝大部分资源投入在工具调用、错误恢复、安全沙箱上，而不是在记忆层上「军备竞赛」。这与当前市场上许多初创公司将「超长上下文」和「永久记忆」作为核心卖点形成对比。

启示：对于 Will 团队来说，OpenClaw 的 L4 已经足够支撑当前业务。与其追求记忆的极致化，不如把资源投入到工具系统的完善（L2）和安全约束的强化（L6）上。

子议题三：Hermes 是竞争者还是组件？

这是本报告最具争议的命题。GLM-5 倾向于将其视为一个需要谨慎评估的外部替代方案，而 Qwen3.5 和 DeepSeek 更倾向于将其定位为 OpenClaw 生态的潜在组件/后端。

视角 将 Hermes 视为竞争者 将 Hermes 视为组件
前提假设 Hermes 会发展为完整 Harness Hermes 专注做记忆层
OpenClaw 的姿态 防御性评估，警惕替代风险 开放性整合，取长补短
最佳策略 保持距离，边缘试点 插件化接口，逐步集成
代表模型 GLM-5 Qwen3.5、DeepSeek

从当前 Hermes 的技术现状来看，它只有 L1/L4 的完整设计，L2/L3/L5/L6 基本上是空白。因此，「组件论」可能更符合现实。但这不排除 Hermes 团队未来向完整 Harness 演进的可能性，届时 GLM-5 的警惕将更有价值。

"共存"复杂度分析

双记忆体系（OpenClaw + Hermes）并存的潜在问题：

  • 故障定位困难：「记忆没找到时，是 Hermes 的问题还是 OpenClaw 的问题？」
  • 数据一致性问题：两份记忆源可能给出矛盾的信息
  • 运维学习成本翻倍：团队需要同时理解两套系统的行为
  • 边际改进有限：当前业务对记忆检索精度的提升感知不明显

GLM-5 的判断是：共存复杂度中等偏高，但得不偿失。DeepSeek 的建议是：可以将其作为「平行层」运行，但不应在核心业务上深度耦合。

投入产出（ROI）分级矩阵

🟢 A 级：维持 OpenClaw 现状

投入最低，产出最稳定。当前业务已验证，无需承担迁移风险。

🟡 B 级：Hermes 做边缘试点

投入中等，用于小范围验证记忆层改进。风险可控，但需要额外运维精力。

🔴 C 级：Hermes 替代 OpenClaw

投入极高，风险极高，回报不确定。需要重建 L2-L6 全套能力，不现实。

5五层耦合分析

DeepSeek 对 OpenClaw × Hermes 替换进行了深入的五层耦合分析。以下是对 L1-L6 每层替换影响的系统评估：

L1 兼容
L2 硬耦合
L3 软耦合
L5 增强
L6 需审查
层级 耦合类型 影响评估
L1 兼容 Hermes 的上下文管理能力与 OpenClaw 的 L1 设计方向一致，增强效应为主。替换 L1 不会破坏上层结构。
L2 硬耦合 这是最大的风险点。Hermes 的工具抽象与 OpenClaw 的 MCP/browser/channel 插件体系可能不兼容。若 Hermes 没有 expose 出对等的工具接口，大量现有 Skills 将无法工作。
L3 软耦合 执行编排层的耦合程度取决于 Hermes 是否提供 subagent/cron/workflow 能力。目前是软耦合，但需要详细的迁移评估。
L4 天然契合 这是 Hermes 的设计原点，理论上可以平滑替换。但需要注意数据迁移和格式兼容问题。
L5 潜在增强 如果 Hermes 能记录失败上下文并提供更好的检索，可能对评估与观测层产生正向增强。但这需要上层系统的配合。
L6 需严格审查 记忆写入不能绕过安全审计层。任何外部记忆系统的引入都必须被纳入 PUA-Lite 的约束框架内，否则可能引入不可控的行为。

💡 核心结论

L2（工具调用）和 L6（安全边界）是 Hermes 集成 OpenClaw 的两个硬门槛。如果这两个问题不解决，任何深度的技术整合都不可行。

6未来演进预测

Hermes 的 6-12 个月可能路径

  1. 路径 A：发展为独立 Harness（概率 20%）

    若团队有充足资金和工程资源，Hermes 可能向 L2-L6 扩展。但考虑到 OpenClaw 已经沉淀了 25+ Skills 和完整的四实例架构，这条路径的投入产出比极低。

  2. 路径 B：作为 OpenClaw 的「记忆后端」（概率 55%）

    最符合 Hermes 技术现状和市场需求的路径。专注做好 L1/L4 的记忆层，通过插件接口或 MCP 协议与上层 Harness 集成。

  3. 路径 C：被收购或并入更大平台（概率 25%）

    若独立发展困难，Hermes/Honcho 的核心技术可能被某个更大的 Agent 平台收购或整合。

OpenClaw 的 6-12 个月可能路径

  1. 短期：L2 和 L6 强化

    继续完善 MCP 生态、浏览器自动化能力，以及 PUA-Lite 约束框架。这是 DeepSeek 和 Qwen3.5 的共同建议。

  2. 中期：L4 检索升级

    在保持现有文件化记忆体系的同时，引入 BM25+rerank 混合检索，向 Hermes 的检索精度看齐。可以借鉴，不一定需要整合。

8术语注释

Harness Engineering
Agent 系统工程的六层模型（L1-L6），涵盖上下文、工具、编排、记忆、观测、约束六个维度。由 Anthropic 和业内学者提出并发展。
Agent
能够感知环境、调用工具、执行多步任务并达成目标的 AI 系统。区别于单次对话的 Model，Agent 强调自主性和持续性。
Model
底层的 LLM（如 GPT-4、Claude、kimi-k2.5、MiniMax M2.7 等），负责语言理解和生成，是 Agent 的「大脑」。
L1-L6
Harness Engineering 的六层架构：L1 上下文边界、L2 工具系统、L3 执行编排、L4 记忆状态、L5 评估观测、L6 约束恢复。
BM25+rerank
一种混合检索技术。BM25 是经典的关键词匹配算法，rerank 是使用神经网络对初筛结果进行二次排序的语义精排步骤。
Context Reflect
让模型主动回顾和整理上下文信息的过程，常用于在长对话或多轮任务中保持信息一致性和准确召回。
MCP
Model Context Protocol，由 Anthropic 提出的开放协议，用于标准化 LLM 与外部工具/数据源之间的交互接口。
PUA-Lite
OpenClaw 采用的行为驱动约束框架，包含三条红线（闭环验证、事实驱动、禁止原地打转）和五步方法论。
Swarm Engine
OpenClaw 的蜂群引擎，通过多 agent 并行执行、独立审查和评分机制，用低端模型数量换高端模型质量。
LCM
Lossless Context Management，OpenClaw 的上下文压缩技术，用于在不丢失关键信息的前提下控制长对话的上下文长度。
