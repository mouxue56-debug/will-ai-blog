🌻 Natsu 详情

角色：SNS 运营 + 内容创作

架构决策 为什么用 Git 而不是云同步？

考虑过：Dropbox / iCloud / Syncthing / GitHub

排除原因：

  • Dropbox/iCloud：无版本控制，冲突处理黑盒
  • Syncthing：局域网可用，但无冲突解决策略
  • GitHub：private repo 有 ToS 风险，且 vault 含敏感数据

选择 Git bare + SSH：完整版本控制 + 自定义冲突策略 + 完全本地

⚡ 延迟：LAN <1ms
🔒 安全：SSH key 认证
04

技术实现深潜

4 个核心模块的完整实现

4.1 Vault → Bare Git Repo

# yuki 上
cd ~/Documents/knowledge-base
git init # 已经有 .git/ (Obsidian Git plugin 建的)
mkdir -p ~/git && git init --bare ~/git/vault.git
git remote add yuki ~/git/vault.git
git push yuki --all
Git 决策 为什么用 bare repo 而不是直接 rsync 工作目录？

bare repo 作为"中央仓库"，各实例的 working tree 独立：

  • 每个实例可以有自己的 uncommitted changes
  • 冲突在 bare 层面解决，不污染 working tree
  • Git 的版本控制提供完整历史回溯

4.2 远程机镜像 bare + clone working

# yuki 上，rsync 推 bare
rsync -aH --delete ~/git/vault.git/ haru:~/git/vault.git/
rsync -aH --delete ~/git/vault.git/ aki:~/git/vault.git/

# 远程机上自己 clone 到 working tree
ssh haru "cd ~/Documents && git clone ~/git/vault.git knowledge-base"
ssh aki "cd ~/Documents && git clone ~/git/vault.git knowledge-base"
为什么不用 GitHub？
vault 里可能有 API keys、决策日志、business data。即使 private repo，云服务有 ToS 风险、账号封禁风险、legal subpoena。自建 bare + Tailscale + SSH 完全本地控制，延迟更低（LAN 毫秒级）。

4.3 clawmem：混合记忆引擎

✓ 双轨索引
→SQLite FTS5（BM25）精确匹配
→LanceDB 向量检索
→Hybrid query 自动加权融合
→本地 embed server（localhost:8088）
✓ 自维护能力
→consolidate 合并低置信重复
→reflect 跨 session 模式分析
→curate 健康检查 + 轻维护
→lifecycle sweep 自动归档
# 各实例 bootstrap
clawmem bootstrap ~/Documents/knowledge-base --name vault
# → 装 Claude Code hooks + MCP server 注册 + 首次索引 1736 docs
索引决策 为什么不同步 LanceDB 索引文件？

LanceDB 索引是二进制文件，跨平台同步问题多：

  • ARM (M4) 和 x86 的向量索引格式不同
  • 索引文件 >100MB，rsync 耗时
  • 每台机的文档子集可能不同（选择性同步）

策略：只同步 markdown 源文件，每台机自己建索引。首次索引 ~2 分钟，后续增量 <10 秒。

4.4 Dream Cycle 脚本核心

#!/bin/bash
# ~/bin/vault-dream-nightly.sh (03:00 via launchd)
set -uo pipefail
VAULT="$HOME/Documents/knowledge-base"
BARE="$HOME/git/vault.git"
CLAWMEM="/opt/homebrew/lib/node_modules/clawmem/bin/clawmem"

# 1. 反向收：让远程 push 本地 working HEAD 到 yuki bare 的临时分支
for HOST in haru aki; do
  ssh "$HOST" "
    cd ~/Documents/knowledge-base
    git add -A
    git diff --staged --quiet || git commit -m 'local changes on $HOST'
    git push yuki HEAD:from-\$HOST --force
  "
done

# 2. yuki merge 临时分支
cd "$VAULT"
git fetch yuki
for HOST in haru aki; do
  git merge --no-edit -X ours "yuki/from-\$HOST" || git merge --abort
  git push yuki --delete "from-\$HOST" 2>/dev/null
done

# 3. clawmem 自维护
"$CLAWMEM" curate
"$CLAWMEM" consolidate
"$CLAWMEM" lifecycle sweep

# 4. yuki 自己的改动 commit + push
git add -A
git diff --staged --quiet || git commit -m "dream cycle: $(date +%F\ %R)"
git push yuki "$BRANCH"

# 5. rsync bare → 远程 bare
for HOST in haru aki; do
  rsync -aH --delete "$BARE/" "$HOST:~/git/vault.git/"
done

# 6. 远程 pull + reindex
for HOST in haru aki; do
  ssh "$HOST" "
    cd ~/Documents/knowledge-base
    git pull --no-rebase
    clawmem update --pull
  "
done
时序决策 为什么选 03:00 而不是 00:00？

00:00 问题：

  • 可能和日期切换的 cron 任务冲突
  • 如果前一天的日志还没写完，会漏同步
  • 网络/电力波动时段（部分 VPS 维护窗口）

03:00 优势：

  • 所有日常任务已完成（日志、备份等）
  • 系统负载最低
  • 如果失败，还有 5 小时可以人工干预（上班前）

踩过的坑 ×7

真实教训，每条都是血泪
🐚 坑 1 macOS /bin/bash 是 3.2（2006年的远古版本）

Apple 2006 年后没升级 /bin/bash（GPL3 license 原因）。很多现代 bash 特性不能用。

# ❌ 不能用
declare -A map  # associative array (bash 4+)
${VAR@Q}         # 引号 quote (bash 4.4+)

# ✅ 替代
brew install bash  # 装 bash 5+
chsh -s /opt/homebrew/bin/bash  # 切换默认 shell
#!/opt/homebrew/bin/bash  # shebang 指定
🐚 坑 2 rsync 删除行为不一致

rsync --delete 在接收端删除文件，但如果源目录结构变化，可能误删。

# ❌ 危险：如果源目录为空，会删光目标
rsync -a --delete src/ dest/

# ✅ 安全：加 --max-delete 限制 + 先 dry-run
rsync -av --delete --max-delete=10 --dry-run src/ dest/
rsync -aH --delete  # -H 保留硬链接（bare repo 需要）
🐚 坑 3 Git merge -X ours 不是 "用我们的"

-X ours 是 "冲突时选我们的"，不是 "完全覆盖"。如果对面新增了文件，还是会合并进来。

# ❌ 误解：以为会完全用 yuki 版本
git merge -X ours remote-branch

# ✅ 真正"只保留我们的"
git merge --strategy=ours remote-branch  # 完全丢弃对方
# 或
git checkout --ours . && git add -A  # 冲突后强制选我们的
🐚 坑 4 LanceDB 跨 ARM/x86 不兼容

M4 (ARM) 和旧 Mac (x86) 的 LanceDB 索引文件不能混用。曾尝试同步索引，结果全量重建。

# ❌ 失败尝试
rsync -a ~/.clawmem/vault.lance/ haru:~/.clawmem/
# → haru 上查询报错：vector dimension mismatch

# ✅ 正确：每台机自己建索引
# yuki 和 haru 分别运行：
clawmem bootstrap ~/Documents/knowledge-base --name vault
# 首次 ~2min，之后增量 <10s
🐚 坑 5 SSH key 权限问题

macOS 升级后 ~/.ssh 权限被重置，导致免密登录失效，Dream Cycle 卡住。

# ❌ 错误权限
ls -la ~/.ssh/id_rsa
# -rw-rw-r--  (group 可写 → SSH 拒绝)

# ✅ 正确权限
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_rsa
chmod 644 ~/.ssh/id_rsa.pub
ssh-add --apple-use-keychain ~/.ssh/id_rsa  # macOS 钥匙串
🐚 坑 6 launchd 环境变量不同

launchd 运行的脚本没有 ~/.zshrc 里的 PATH，导致 clawmem 命令找不到。

# ❌ 失败：假设有 PATH
clawmem curate  # launchd: command not found

# ✅ 方案 A：写绝对路径
CLAWMEM="/opt/homebrew/lib/node_modules/clawmem/bin/clawmem"
"$CLAWMEM" curate

# ✅ 方案 B：launchd plist 里加 EnvironmentVariables
<key>EnvironmentVariables</key>
<dict>
  <key>PATH</key>
  <string>/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin</string>
</dict>
🐚 坑 7 Obsidian Git 插件和脚本冲突

Obsidian Git 插件自动 commit，和 Dream Cycle 脚本同时操作，导致 conflict。

# ❌ 冲突场景
# 03:00 Obsidian Git 自动 commit "vault backup: 2026-04-20"
# 03:00 Dream Cycle 脚本同时 commit "dream cycle: 2026-04-20 03:00"
# → Git lock 冲突

# ✅ 解决：Obsidian Git 改到 02:30，留 30min 缓冲
# 或完全禁用自动 commit，只用 Dream Cycle

观感与收获

半年实践的真实体会
🎯
决策可追溯
每个技术选型都有"为什么"记录，三个月后回看不会懵。Git blame 知道是谁、什么时候、为什么做的决定。
🔄
知识自动沉淀
重要的对话、决策、踩坑，自动归档到 Vault。不用手动整理，clawmem 会帮你分类。
🚀
onboarding 加速
新实例（或新模型）接入时，读一遍 Vault 就能了解全部上下文，不用从头教。

最大的收获是"忘记"的自由。以前每个细节都要记，现在知道"写进 Vault 就行"，大脑可以专注思考而不是记忆。

如果你也想建共享大脑

1
建一个 Git repo，放 markdown 文件
2
各实例配好 Git 同步（cron 或手动）

先用起来，别追求完美结构

✗
不要同步二进制索引 → 源文件足够，索引自己建
✗

决策树 你需要共享大脑吗？

✓ 需要：多个 AI 实例 / 跨设备工作 / 长期项目 / 团队协作

✗ 不需要：单个 ChatGPT 账号 / 纯临时任务 / 不重复的工作流

临界点：当你发现"这个我上次好像解决过"超过 3 次，就该建了。
